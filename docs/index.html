Regra principal

Antes de editar, descubra o dono do comportamento. Neste projeto, quase todo bug e quase toda feature ficam faceis quando voce acerta o arquivo dono.

Comece aqui

Abra o projeto sem medo

Voce nao precisa dominar C++, nem entender todo o engine, nem conhecer arquitetura avancada de Unity para mexer no VoxelParadox.Client. O caminho seguro e: entender o fluxo alto nivel, achar o dono do sistema e editar o ponto certo.

Caminho fisico vs include

No disco voce vai ver caminhos como src/World/world/world_stack.hpp. Mas os includes do projeto continuam simples, por exemplo #include "world/world_stack.hpp". Pense assim: a estrutura fisica organiza o projeto; os aliases de include mantem o codigo limpo.

Fluxo do jogo

O caminho do jogo do exe ate um frame na tela

1. main()

O executavel entra em src/Core/runtime/main.cpp e chama runRuntimeApp().

2. Bootstrap

Settings, presets de biome, janela, renderer, HUD, mundo raiz, player e audio sobem antes do loop.

3. Mundo ativo

WorldStack cria e controla o mundo atual. FractalWorld cuida dos chunks.

4. Loop principal

Input, gameplay, mundo, audio, render 3D, HUD e swap buffers acontecem em ordem fixa por frame.

main()
  -> runRuntimeApp()
      -> load settings
      -> load biome presets
      -> create window
      -> init renderer
      -> init HUD
      -> create root world
      -> create player
      -> start main loop
main loop
  -> Input::update()
  -> chat + atalhos globais
  -> player.update()
  -> worldStack.update()
  -> audioController.syncFrame()
  -> renderer.render()
  -> HUD::update()
  -> HUD::render()
  -> glfwSwapBuffers()

C++ basico

O minimo de C++ que voce precisa para trabalhar aqui

.hpp

Pense como a face publica da classe: o que ela tem e o que ela sabe fazer.

.cpp

Pense como a parte que realmente executa o comportamento.

T&

Referencia. A funcao recebe o objeto real. Se editar, edita o original.

T*

Ponteiro. Pode ser nulo. Normalmente significa "talvez exista".

const

Leia como "nao modificar por aqui". E parte do contrato da funcao.

enum class

Lista de estados nomeados, como PlayerTransition.

Jeito simples de ler uma assinatura

void update(float dt, WorldStack& worldStack)
  • void: nao retorna valor.
  • update: nome da acao.
  • float dt: recebe tempo do frame.
  • WorldStack&: recebe o mundo real, nao uma copia.

Regras de edicao

Como editar sem criar bagunca

1. Descubra o dono

Nao edite o primeiro arquivo que voce viu. Edite quem manda na regra.

2. Evite hardcode

Defaults vao para client_defaults.hpp. Paths vao para client_assets.hpp.

3. Persistencia correta

Se a mudanca precisa continuar existindo, ela deve passar pelo caminho de persistencia do mundo.

4. Use helpers semanticos

Regras como "e solido?" ou "pode mirar?" devem viver em helpers, nao copiadas por varios arquivos.

5. Pense em projeto tambem

Se criar arquivo fonte novo, o Visual Studio precisa conhecer esse arquivo no .vcxproj e no .filters.

6. Teste o caminho feliz

Depois de compilar, teste pelo menos a acao principal que voce mexeu.

Regra pratica de mundo

Mudar o array do chunk carregado nem sempre basta. Quando a mudanca e "de verdade", ela precisa entrar pelo caminho que o mundo salva e reaplica depois.

Tarefas comuns

Receitas curtas para modificacoes frequentes

Quero mexer no movimento do player
  1. Abra src/Entities/player/player_movement.cpp.
  2. Procure por aceleracao, drag, sprint, colisao ou camera look.
  3. Compile e teste no jogo. Mudancas aqui costumam ser faceis de validar.
Quero mudar como quebrar ou colocar blocos funciona
  1. Abra src/Entities/player/player_interaction.cpp.
  2. Procure o fluxo de raycast, validacao do alvo, progresso e conclusao.
  3. Se a regra for de bloco, mova a parte semantica para helpers de bloco ou item.
Quero mudar HUD, menu ou texto de interface
  1. Montagem de HUD: src/Core/runtime/runtime_ui_hud.cpp.
  2. Atalhos e estado de UI: src/Core/runtime/runtime_ui.cpp.
  3. Widgets base: src/Renderer/render/hud/*.
Quero adicionar item ou receita
  1. Item: src/Entities/items/item_catalog.hpp.
  2. Crafting: src/Entities/player/crafting.*.
  3. Dados de receita: res/Assets/Recipes/.
Quero mudar geracao de mundo ou biomes
  1. Geracao: src/World/world/chunk_generator_source.*.
  2. Presets de biome: src/World/world/biome_preset.*.
  3. Pilha e persistencia: src/World/world/world_stack.*.
Quero mudar render, preview ou modelo 3D
  1. Render geral: src/Renderer/render/renderer.cpp.
  2. Passes auxiliares: renderer_items.cpp, renderer_portals.cpp, renderer_entities.cpp.
  3. Assets de modelo: src/Renderer/entities e res/Assets/Models/.

Checklist final

Antes de fechar uma task

Compila?

Se nao compila, o resto ainda nao importa.

Foi no arquivo certo?

Se o bug continua igual, talvez a regra esteja em outro dono.

Sem hardcode novo?

Paths, ids e defaults deveriam estar centralizados.

Persiste quando precisa?

Especialmente para mundo, blocos, inventario e saves.

Projeto atualizado?

Se criou arquivo novo, confirme .vcxproj e .filters.

Teste rapido no jogo?

Pelo menos o caminho principal da mudanca precisa rodar de ponta a ponta.

Esta documentacao foi escrita para ser pratica. Se o codigo mudar, atualize primeiro este mapa mental e depois aprofunde os detalhes.
